# 第 8 章 网络编程

## 8.1 Socket 是什么

Socket（套接字）是内核抽象的一种**通信端点**，可以理解为一个双向的管道 fd。`socket()` 系统调用创建它，之后用 `read/write/send/recv` 读写，用 `close` 销毁——和普通文件 fd 一样，这正是"一切皆文件"的体现。

网络栈的分层模型对应到 Linux 的实现：

```
应用层       read/write/send/recv        ← 用户态调用
─────────────────────────────────────────
传输层       TCP (SOCK_STREAM)            ← 内核 tcp_sendmsg / tcp_recvmsg
             UDP (SOCK_DGRAM)             ← 内核 udp_sendmsg / udp_recvmsg
─────────────────────────────────────────
网络层       IP 路由、分片               ← 内核 ip_output / ip_input
─────────────────────────────────────────
链路层       以太网帧、ARP               ← 网卡驱动
─────────────────────────────────────────
物理层       电信号 / 光信号
```

## 8.2 socket / bind / listen / accept / connect

```c
#include <sys/socket.h>
int socket(int domain, int type, int protocol);
int bind  (int sockfd, const struct sockaddr *addr, socklen_t addrlen);
int listen(int sockfd, int backlog);
int accept(int sockfd, struct sockaddr *addr, socklen_t *addrlen);
int accept4(int sockfd, struct sockaddr *addr, socklen_t *addrlen, int flags);
int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
```

| 参数 | 常用值 |
| --- | --- |
| domain | `AF_INET`（IPv4）、`AF_INET6`（IPv6）、`AF_UNIX`（本机） |
| type | `SOCK_STREAM`（TCP）、`SOCK_DGRAM`（UDP）、`SOCK_RAW` |
| type 附加 | `\| SOCK_NONBLOCK \| SOCK_CLOEXEC`（Linux 2.6.27+，推荐） |
| protocol | 通常 0（内核自动选） |

### TCP 服务端连接建立的内核流程

```
三次握手在内核完成，用户态只需 accept()

client: connect()  → SYN →
server 内核:       ← SYN-ACK ← （放入 SYN 队列，半连接）
client: connect 返回0 → ACK →
server 内核:       ACK 收到 → 移入 accept 队列（全连接队列）
server: accept()   → 从全连接队列取出 → 返回新 fd
```

**backlog**（`listen` 第二参数）控制的是**全连接队列**的最大长度（Linux 内核 2.2 之后）。SYN 队列大小由 `/proc/sys/net/ipv4/tcp_max_syn_backlog` 控制。

```bash
ss -lnt   # 查看监听中的 socket 的 backlog 使用情况（Recv-Q）
```

### 完整 TCP 回声服务器

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <errno.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <sys/socket.h>

#define PORT 8888
#define BUF  4096

static void handle_client(int cfd, struct sockaddr_in *peer)
{
    char client_ip[INET_ADDRSTRLEN];
    inet_ntop(AF_INET, &peer->sin_addr, client_ip, sizeof(client_ip));
    printf("连接来自 %s:%d\n", client_ip, ntohs(peer->sin_port));

    char buf[BUF];
    ssize_t n;
    while ((n = recv(cfd, buf, sizeof(buf), 0)) > 0) {
        send(cfd, buf, (size_t)n, 0);   /* echo */
    }
    close(cfd);
    printf("%s 断开\n", client_ip);
}

int main(void)
{
    int lfd = socket(AF_INET, SOCK_STREAM | SOCK_CLOEXEC, 0);

    /* SO_REUSEADDR：TIME_WAIT 期间允许重新 bind 同一端口 */
    int yes = 1;
    setsockopt(lfd, SOL_SOCKET, SO_REUSEADDR, &yes, sizeof(yes));

    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_port   = htons(PORT),
        .sin_addr.s_addr = INADDR_ANY,
    };
    bind(lfd, (struct sockaddr *)&addr, sizeof(addr));
    listen(lfd, 128);
    printf("监听 0.0.0.0:%d\n", PORT);

    for (;;) {
        struct sockaddr_in peer;
        socklen_t plen = sizeof(peer);
        int cfd = accept4(lfd, (struct sockaddr *)&peer, &plen, SOCK_CLOEXEC);
        if (cfd < 0) { perror("accept4"); continue; }
        handle_client(cfd, &peer);   /* 单进程单连接演示，实际要多线程/多进程 */
    }
}
```

## 8.3 字节序与地址转换

网络传输使用**大端（Big Endian）字节序**，x86 是**小端**，所以多字节数值（端口号、IP 地址）在网络传输前必须转换：

```c
#include <arpa/inet.h>
uint16_t htons(uint16_t host);   /* host → network，16 位 */
uint32_t htonl(uint32_t host);   /* host → network，32 位 */
uint16_t ntohs(uint16_t net);    /* network → host，16 位 */
uint32_t ntohl(uint32_t net);    /* network → host，32 位 */

/* IP 地址字符串 <-> 二进制（AF_INET / AF_INET6 通用） */
int         inet_pton(int af, const char *src, void *dst);
const char *inet_ntop(int af, const void *src, char *dst, socklen_t size);
```

```c
struct sockaddr_in6 addr6 = { .sin6_family = AF_INET6 };
inet_pton(AF_INET6, "::1", &addr6.sin6_addr);   /* IPv6 回环 */
```

## 8.4 TCP 的重要特性与选项

### 粘包问题

TCP 是**字节流**，不保留消息边界。发送方发了两条消息（`"hello"` + `"world"`），接收方可能一次 `recv` 全收到，也可能分两次收，甚至拼在一起。

**解决粘包的三种方案**：

- 固定长度消息：每次 recv 读固定字节
- 定界符：用 `\n` 或 `\r\n` 分隔（HTTP 头部就是这样）
- 长度前缀：消息头 4 字节记录后续长度，先读 4 字节，再按长度读内容（最通用）

```c
/* 长度前缀协议：发送 */
uint32_t len_be = htonl((uint32_t)msglen);
send(fd, &len_be, 4, MSG_MORE);       /* MSG_MORE 告知内核还有数据，合并发送 */
send(fd, msg, msglen, 0);

/* 接收 */
uint32_t len_be;
read_full(fd, &len_be, 4);
uint32_t len = ntohl(len_be);
char *buf = malloc(len + 1);
read_full(fd, buf, len);
```

### 常用 socket 选项

```c
setsockopt(fd, level, optname, &val, sizeof(val));
getsockopt(fd, level, optname, &val, &len);
```

| level | optname | 作用 |
| --- | --- | --- |
| `SOL_SOCKET` | `SO_REUSEADDR` | 允许 TIME_WAIT 期间重用端口（服务器必加） |
| `SOL_SOCKET` | `SO_REUSEPORT` | 多进程/线程可 bind 同一端口（负载均衡） |
| `SOL_SOCKET` | `SO_KEEPALIVE` | 开启 TCP 保活（长连接检测对端存活） |
| `SOL_SOCKET` | `SO_SNDBUF / SO_RCVBUF` | 调整发送/接收缓冲区大小 |
| `SOL_SOCKET` | `SO_LINGER` | close 时等待数据发送完毕 |
| `IPPROTO_TCP` | `TCP_NODELAY` | 关闭 Nagle 算法（低延迟场景必加） |
| `IPPROTO_TCP` | `TCP_CORK` | 积攒数据后一起发（与 MSG_MORE 类似） |
| `IPPROTO_TCP` | `TCP_KEEPIDLE` | 保活探测开始前的空闲时间（秒） |
| `IPPROTO_TCP` | `TCP_KEEPINTVL` | 两次探测间隔（秒） |
| `IPPROTO_TCP` | `TCP_KEEPCNT` | 探测次数，全失败则断开 |
| `IPPROTO_TCP` | `TCP_FASTOPEN` | 快速打开（TFO），减少一次 RTT |

### Nagle 算法与 TCP_NODELAY

Nagle 算法：有未确认数据时，将小包积攒到缓冲区合并发送，提高带宽利用率。副作用是增加延迟（最多 200ms）。

**实时应用（游戏、远程控制）必须关 Nagle**：

```c
int flag = 1;
setsockopt(fd, IPPROTO_TCP, TCP_NODELAY, &flag, sizeof(flag));
```

### TIME_WAIT 状态

主动关闭 TCP 连接的一方会进入 TIME_WAIT，持续 2×MSL（默认约 60 秒），此时端口不可重用。服务器重启时常见 `bind: Address already in use`。

解决方案：

```c
int yes = 1;
setsockopt(lfd, SOL_SOCKET, SO_REUSEADDR, &yes, sizeof(yes));
/* Linux 4.6+ 可以用 SO_REUSEPORT 多进程负载均衡 */
```

## 8.5 UDP 编程

UDP 是无连接的数据报协议，不保证顺序、不重传、不建立连接，但延迟低、开销小。

```c
#include <sys/socket.h>
ssize_t sendto  (int fd, const void *buf, size_t len, int flags,
                 const struct sockaddr *dest, socklen_t addrlen);
ssize_t recvfrom(int fd, void *buf, size_t len, int flags,
                 struct sockaddr *src_addr, socklen_t *addrlen);
```

### UDP 服务端

```c
int fd = socket(AF_INET, SOCK_DGRAM | SOCK_CLOEXEC, 0);
struct sockaddr_in addr = { .sin_family=AF_INET, .sin_port=htons(9999), .sin_addr.s_addr=INADDR_ANY };
bind(fd, (struct sockaddr *)&addr, sizeof(addr));

char buf[1500];  /* 不超过以太网 MTU，避免 IP 分片 */
struct sockaddr_in peer;
socklen_t plen = sizeof(peer);
ssize_t n = recvfrom(fd, buf, sizeof(buf), 0, (struct sockaddr *)&peer, &plen);
sendto(fd, buf, (size_t)n, 0, (struct sockaddr *)&peer, plen);  /* echo */
```

UDP 的适用场景：DNS 查询、实时音视频（允许丢包）、广播/组播、QUIC（在 UDP 上自实现可靠传输）。

## 8.6 getaddrinfo —— 现代域名解析

`gethostbyname` 不支持 IPv6 且线程不安全，已废弃。用 `getaddrinfo`：

```c
#include <netdb.h>
int getaddrinfo(const char *node, const char *service,
                const struct addrinfo *hints, struct addrinfo **res);
void freeaddrinfo(struct addrinfo *res);
const char *gai_strerror(int ecode);
```

```c
struct addrinfo hints = {
    .ai_family   = AF_UNSPEC,       /* IPv4 or IPv6 */
    .ai_socktype = SOCK_STREAM,     /* TCP */
};
struct addrinfo *res;
int rc = getaddrinfo("example.com", "80", &hints, &res);
if (rc) { fprintf(stderr, "%s\n", gai_strerror(rc)); exit(1); }

for (struct addrinfo *p = res; p; p = p->ai_next) {
    int fd = socket(p->ai_family, p->ai_socktype | SOCK_CLOEXEC, p->ai_protocol);
    if (connect(fd, p->ai_addr, p->ai_addrlen) == 0) {
        /* 连接成功，用 fd */
        break;
    }
    close(fd);
}
freeaddrinfo(res);
```

## 8.7 IO 多路复用：select / poll / epoll

当一个进程/线程需要同时处理多个 socket 时，不能为每个 socket 都阻塞等待，需要 IO 多路复用：**谁有数据谁先处理**。

### select（历史接口，了解即可）

```c
#include <sys/select.h>
int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds,
           struct timeval *timeout);
FD_SET(fd, &set);  FD_CLR(fd, &set);  FD_ISSET(fd, &set);  FD_ZERO(&set);
```

缺点：fd 上限 1024，每次调用都要把整个 fd_set 从用户态复制到内核，O(n) 扫描，不适合大量连接。

### poll（稍好，但仍有 O(n) 问题）

```c
#include <poll.h>
struct pollfd { int fd; short events; short revents; };
int poll(struct pollfd *fds, nfds_t nfds, int timeout_ms);
```

没有 1024 fd 上限，但每次仍需复制整个数组，O(n) 扫描。

### epoll（Linux 特有，高性能）

epoll 的核心改进：

- 用户只需一次 `epoll_ctl` 注册 fd，**无需每次 call 重新传递 fd 列表**
- 内核只返回**就绪的 fd**，不扫描整个集合，O(1) 返回
- 内部用**红黑树**存储注册的 fd，**链表**维护就绪列表

```c
#include <sys/epoll.h>
int epoll_create1(int flags);                          /* flags: EPOLL_CLOEXEC */
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *ev);
/* op: EPOLL_CTL_ADD / EPOLL_CTL_MOD / EPOLL_CTL_DEL */
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout_ms);
```

`epoll_event.events` 常用标志：

| 标志 | 含义 |
| --- | --- |
| `EPOLLIN` | fd 可读（有数据或对端关闭） |
| `EPOLLOUT` | fd 可写（发送缓冲区有空间） |
| `EPOLLRDHUP` | 对端关闭写端（半关闭） |
| `EPOLLERR` | 错误（无需注册，始终监测） |
| `EPOLLHUP` | 挂起（对端关闭，无需注册） |
| `EPOLLET` | 边缘触发（Edge Triggered） |
| `EPOLLONESHOT` | 事件触发一次后自动从 epoll 移除（需重新 EPOLL_CTL_MOD 才能继续） |

### 水平触发（LT）vs 边缘触发（ET）

| 模式 | 触发条件 | 特点 |
| --- | --- | --- |
| LT（默认） | 只要 fd 就绪就触发 | 类似 select/poll，较简单，允许不一次读完 |
| ET | 状态从"未就绪"变为"就绪"时触发一次 | 必须一次读完所有数据，必须配合非阻塞 fd，效率更高 |

**ET 模式下读数据的正确姿势**：

```c
/* ET 模式：必须循环读到 EAGAIN */
while (1) {
    ssize_t n = recv(fd, buf, sizeof(buf), 0);
    if (n < 0) {
        if (errno == EAGAIN || errno == EWOULDBLOCK) break;  /* 读完了 */
        /* 真正的错误 */
        close_connection(fd);
        break;
    }
    if (n == 0) { close_connection(fd); break; }  /* 对端关闭 */
    process(buf, n);
}
```

### 完整 epoll 事件循环示例

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <errno.h>
#include <netinet/in.h>
#include <sys/socket.h>
#include <sys/epoll.h>

#define PORT    8888
#define MAXEVS  64
#define BUFSIZE 4096

static void set_nonblocking(int fd)
{
    int flags = fcntl(fd, F_GETFL);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main(void)
{
    /* 创建监听 socket */
    int lfd = socket(AF_INET, SOCK_STREAM | SOCK_CLOEXEC | SOCK_NONBLOCK, 0);
    int yes = 1;
    setsockopt(lfd, SOL_SOCKET, SO_REUSEADDR, &yes, sizeof(yes));
    struct sockaddr_in addr = {
        .sin_family = AF_INET, .sin_port = htons(PORT),
        .sin_addr.s_addr = INADDR_ANY
    };
    bind(lfd, (struct sockaddr *)&addr, sizeof(addr));
    listen(lfd, 128);

    /* 创建 epoll 实例 */
    int epfd = epoll_create1(EPOLL_CLOEXEC);
    struct epoll_event ev = { .events = EPOLLIN, .data.fd = lfd };
    epoll_ctl(epfd, EPOLL_CTL_ADD, lfd, &ev);

    struct epoll_event evs[MAXEVS];
    char buf[BUFSIZE];
    for (;;) {
        int n = epoll_wait(epfd, evs, MAXEVS, -1);  /* -1 = 永久等待 */
        for (int i = 0; i < n; i++) {
            int fd = evs[i].data.fd;
            if (fd == lfd) {
                /* 有新连接 */
                struct sockaddr_in peer;
                socklen_t plen = sizeof(peer);
                int cfd = accept4(lfd, (struct sockaddr *)&peer, &plen,
                                  SOCK_NONBLOCK | SOCK_CLOEXEC);
                if (cfd < 0) continue;
                ev.events  = EPOLLIN | EPOLLET;  /* ET 模式 */
                ev.data.fd = cfd;
                epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &ev);
            } else if (evs[i].events & (EPOLLERR | EPOLLHUP)) {
                epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
                close(fd);
            } else if (evs[i].events & EPOLLIN) {
                /* ET 模式：循环读完 */
                while (1) {
                    ssize_t nr = recv(fd, buf, sizeof(buf), 0);
                    if (nr < 0) {
                        if (errno == EAGAIN) break;
                        close(fd); break;
                    }
                    if (nr == 0) { close(fd); break; }
                    send(fd, buf, (size_t)nr, 0);  /* echo */
                }
            }
        }
    }
}
```

## 8.8 非阻塞 connect 与连接超时

```c
set_nonblocking(fd);
int rc = connect(fd, addr, addrlen);
if (rc < 0 && errno != EINPROGRESS) { /* 真正的错误 */ }

/* 用 epoll/poll 等待 fd 可写，表示连接完成 */
struct pollfd pfd = { .fd = fd, .events = POLLOUT };
poll(&pfd, 1, 5000);   /* 5 秒超时 */

/* 检查连接是否成功 */
int err;
socklen_t len = sizeof(err);
getsockopt(fd, SOL_SOCKET, SO_ERROR, &err, &len);
if (err) { errno = err; /* 连接失败 */ }
```

## 8.9 send / recv 的标志

```c
ssize_t send(int fd, const void *buf, size_t len, int flags);
ssize_t recv(int fd, void *buf, size_t len, int flags);
```

| flag | 含义 |
| --- | --- |
| `MSG_WAITALL` | 接收方等所有字节到齐再返回（TCP 流的"完整读"） |
| `MSG_DONTWAIT` | 本次调用非阻塞（不修改 fd 的全局 O_NONBLOCK） |
| `MSG_PEEK` | 读取但不消耗数据（数据还在缓冲区，下次 recv 仍能读到） |
| `MSG_NOSIGNAL` | 写断开的连接时不发 SIGPIPE，返回 EPIPE（推荐） |
| `MSG_MORE` | 告知内核后面还有数据，延迟发送（类似 TCP_CORK） |
| `MSG_OOB` | 紧急数据（带外数据，已基本废弃） |

## 8.10 sendfile —— 零拷贝文件发送

传统 `read + write` 发送文件要经历 4 次拷贝：磁盘 → 内核 page cache → 用户缓冲区 → 内核 socket 缓冲区 → 网卡。

`sendfile` 在内核内部完成，省掉了用户态的两次拷贝：

```c
#include <sys/sendfile.h>
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
```

```c
struct stat st;
fstat(file_fd, &st);
/* offset=NULL 从文件当前位置发，否则从 *offset 发并更新 offset */
sendfile(sock_fd, file_fd, NULL, (size_t)st.st_size);
```

HTTP 静态文件服务器的核心就是这一行。Nginx 默认开启 `sendfile on`。

更进一步的零拷贝：`splice`（在两个 fd 间移动数据，不经过用户态）、`tee`（复制管道数据）。

## 8.11 连接复用模型对比

| 模型 | 典型代表 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 每连接一线程 | 早期 Java 服务器 | 代码简单 | 线程数 = 连接数，C10K 后不可行 |
| 每连接一进程 | Apache prefork | 隔离好 | 内存占用高 |
| Reactor（事件驱动） | Nginx / Redis / Node.js | 单线程/少线程处理大量连接 | CPU 密集任务会阻塞事件循环 |
| 多 Reactor | Nginx worker 多进程 | 充分利用多核 | 代码复杂 |
| 半同步半异步 | 线程池 + 队列 | 兼顾 IO 和 CPU | 上下文切换开销 |
| io_uring | Linux 5.1+ | 真正异步，接近零系统调用 | 较新，生产环境逐步普及 |

## 8.12 io_uring 简介

`io_uring` 是 Linux 5.1 引入的新型异步 IO 接口，通过共享内存环形队列（ring buffer）在用户态和内核态之间传递请求和结果，极端情况下**一次系统调用都不用**就能完成 IO：

```c
#include <liburing.h>    /* 需要安装 liburing，链接 -luring */

struct io_uring ring;
io_uring_queue_init(256, &ring, 0);  /* 深度 256 的队列 */

/* 提交一个 read 请求 */
struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, sizeof(buf), 0);
io_uring_sqe_set_data(sqe, (void *)(uintptr_t)fd);
io_uring_submit(&ring);

/* 等待完成 */
struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);
printf("读了 %d 字节\n", cqe->res);
io_uring_cqe_seen(&ring, cqe);
io_uring_queue_exit(&ring);
```

## 8.13 易错点汇总

- **recv 返回 0 没有当 EOF 处理**：TCP 对端关闭连接，recv 返回 0，不是错误，应关闭 fd
- **粘包没处理**：TCP 流，直接 recv 当消息用，必然出数据错位
- **端口号没 htons**：`sin_port = 8080` 而不是 `htons(8080)`，在大端机器上 bind 到了别的端口
- **没加 SO_REUSEADDR**：服务器重启 bind 失败 EADDRINUSE
- **epoll ET 模式没循环读完**：只读一次 recv，下次再也不会被通知（ET 只触发一次变化）
- **epoll ET 模式的 fd 不是非阻塞**：循环读完后 recv 阻塞，事件循环死锁
- **多线程共享一个 epoll fd 的惊群**：所有线程同时被唤醒，只有一个能 accept，其他白跑（用 EPOLLEXCLUSIVE 或 SO_REUSEPORT 解决）
- **sendfile 发完没检查返回值**：网络不稳时可能短发，需要循环
- **connect 非阻塞后直接读写**：没等 POLLOUT 就 send，数据丢失
