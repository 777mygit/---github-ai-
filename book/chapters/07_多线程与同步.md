# 第 7 章 多线程与同步

## 7.1 线程 vs 进程：为什么选线程

从内核视角，线程是共享同一地址空间的 `task_struct`。用 `clone(CLONE_VM | CLONE_FILES | CLONE_SIGHAND | ...)` 创建，比 `fork` 轻量，但也带来了"共享"的双刃剑：

| 维度 | 多进程 | 多线程 |
| --- | --- | --- |
| 地址空间 | 独立（fork COW） | 共享 |
| 数据共享 | 需要 IPC | 直接读写共享变量 |
| 创建开销 | 较高（复制页表） | 极低 |
| 崩溃隔离 | 一个崩溃不影响其他 | 一个线程崩溃 → 整个进程崩溃 |
| 同步复杂度 | 低（数据不共享） | 高（竞态、死锁） |
| 适合场景 | 独立任务、强隔离要求 | 高并发 IO、共享大数据集 |

**线程适合的场景**：

- 大量并发 IO（网络服务器：每个连接一个线程）
- 并行计算（多核共享内存数组，加速处理）
- UI + 后台任务分离（主线程 UI，工作线程下载）

## 7.2 创建和管理线程

```c
#include <pthread.h>
/* 编译时加 -pthread */

int  pthread_create(pthread_t *tid, const pthread_attr_t *attr,
                    void *(*start)(void *), void *arg);
int  pthread_join  (pthread_t tid, void **retval);
int  pthread_detach(pthread_t tid);
void pthread_exit  (void *retval);
int  pthread_cancel(pthread_t tid);
pthread_t pthread_self(void);
int  pthread_equal(pthread_t t1, pthread_t t2);
```

### 线程函数签名必须是 `void *f(void *)`

```c
typedef struct { int start; int end; long *result; } TaskArg;

static void *sum_range(void *arg)
{
    TaskArg *t = arg;
    long s = 0;
    for (int i = t->start; i <= t->end; i++) s += i;
    *t->result = s;
    return NULL;   /* 或 pthread_exit(NULL) */
}

int main(void)
{
    long r1 = 0, r2 = 0;
    TaskArg a1 = { 1,    500000, &r1 };
    TaskArg a2 = { 500001, 1000000, &r2 };
    pthread_t t1, t2;

    pthread_create(&t1, NULL, sum_range, &a1);
    pthread_create(&t2, NULL, sum_range, &a2);

    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    printf("1..1000000 之和 = %ld\n", r1 + r2);
    return 0;
}
```

### join vs detach

| 状态 | 含义 | 后果 |
| --- | --- | --- |
| joinable（默认） | 退出后保留退出状态，等待被 join | 不 join → 类似僵尸，资源不释放 |
| detached | 退出后内核自动回收 | 不能 join，适合"发射后不管"的后台线程 |

```c
pthread_detach(pthread_self());   /* 线程自己 detach */

/* 或者创建时设属性 */
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_DETACHED);
pthread_create(&tid, &attr, func, arg);
pthread_attr_destroy(&attr);
```

### pthread_cancel —— 取消线程

```c
int pthread_cancel(pthread_t tid);            /* 发送取消请求 */
int pthread_setcancelstate(int state, int *old);  /* PTHREAD_CANCEL_ENABLE / DISABLE */
int pthread_setcanceltype (int type,  int *old);  /* PTHREAD_CANCEL_DEFERRED / ASYNCHRONOUS */
void pthread_testcancel(void);                /* 手动设置取消点 */
```

**取消点**是线程检查取消请求的地方（`read/write/sleep/pthread_cond_wait` 等都是取消点）。`PTHREAD_CANCEL_DEFERRED` 模式下，取消只在取消点发生，相对安全；`ASYNCHRONOUS` 模式下随时取消，极不安全（锁可能被持有中）。

取消时可通过 **cleanup handler** 保证资源释放：

```c
static void cleanup(void *arg)
{
    printf("cleanup: 释放资源\n");
    free(arg);
}

void *worker(void *arg)
{
    void *buf = malloc(1024);
    pthread_cleanup_push(cleanup, buf);   /* 注册清理函数 */

    /* ... 可能被取消的代码 ... */
    sleep(10);   /* 取消点：如果被 cancel，在这里跳出去执行 cleanup */

    pthread_cleanup_pop(1);   /* 1 = 执行并弹出；0 = 只弹出不执行 */
    return NULL;
}
```

## 7.3 互斥锁（pthread_mutex）

互斥锁保证同一时刻只有一个线程进入临界区。从内核视角，`pthread_mutex` 在用户态用**原子操作（CAS）**实现快路径，只有竞争时才通过 **futex** 系统调用陷入内核等待，所以无竞争时开销极低（约几十纳秒）。

### futex（Fast Userspace Mutex）原理

```
未竞争：
  线程 A: CAS(mutex, 0, 1) 成功 → 直接获得锁，不进内核

有竞争：
  线程 A 持有锁（mutex = 1）
  线程 B: CAS(mutex, 0, 1) 失败 → 设 mutex = 2（表示有等待者）
          → syscall futex(FUTEX_WAIT, mutex_addr, 2) 陷入内核睡眠
  线程 A 释放: 检查 mutex == 2 → syscall futex(FUTEX_WAKE) 唤醒 B
```

### API 与使用模板

```c
#include <pthread.h>

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;  /* 静态初始化 */

/* 动态初始化 */
pthread_mutex_t mutex2;
pthread_mutex_init(&mutex2, NULL);   /* 第二参数 = 属性，NULL 为默认 */

pthread_mutex_lock(&mutex);          /* 获取锁，阻塞等待 */
pthread_mutex_trylock(&mutex);       /* 尝试获取，失败返回 EBUSY */
pthread_mutex_timedlock(&mutex, &abs_timeout);  /* 带超时 */
pthread_mutex_unlock(&mutex);        /* 释放锁 */
pthread_mutex_destroy(&mutex);       /* 销毁 */
```

### 互斥锁的类型（pthread_mutexattr）

| 类型 | 行为 |
| --- | --- |
| `PTHREAD_MUTEX_NORMAL` | 默认，不检测死锁，重复 lock 同一把锁 → 死锁 |
| `PTHREAD_MUTEX_ERRORCHECK` | 重复 lock 或非属主 unlock 返回错误 |
| `PTHREAD_MUTEX_RECURSIVE` | 允许同一线程重复 lock，需对应次数 unlock |
| `PTHREAD_MUTEX_DEFAULT` | 实现定义，Linux 上等同 NORMAL |

```c
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
pthread_mutex_t mu;
pthread_mutex_init(&mu, &attr);
pthread_mutexattr_destroy(&attr);
```

### 计数器示例（无锁 → 有锁）

```c
#define _GNU_SOURCE
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>

#define N_THREADS 8
#define N_INC     1000000

static long counter = 0;
static pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;

static void *inc(void *arg)
{
    for (int i = 0; i < N_INC; i++) {
        pthread_mutex_lock(&mu);
        counter++;           /* 临界区 */
        pthread_mutex_unlock(&mu);
    }
    return NULL;
}

int main(void)
{
    pthread_t tids[N_THREADS];
    for (int i = 0; i < N_THREADS; i++)
        pthread_create(&tids[i], NULL, inc, NULL);
    for (int i = 0; i < N_THREADS; i++)
        pthread_join(tids[i], NULL);
    printf("counter = %ld (期望 %ld)\n", counter, (long)N_THREADS * N_INC);
    return 0;
}
```

## 7.4 读写锁（pthread_rwlock）

互斥锁对读操作太保守——多个线程**同时读**同一数据是安全的，不需要互斥。读写锁允许：

- **多个读者同时持锁**（共享读）
- **只有一个写者持锁，且此时没有读者**（独占写）

```
读者 1 ──►  [RLock]  ──► 读
读者 2 ──►  [RLock]  ──► 读   （同时进行，没问题）
写者   ──►  [WLock]  ──► 阻塞，等所有读者释放后才能进入
```

```c
pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

pthread_rwlock_rdlock(&rwlock);   /* 获取读锁 */
pthread_rwlock_tryrdlock(&rwlock);
/* ... 读操作 ... */
pthread_rwlock_unlock(&rwlock);

pthread_rwlock_wrlock(&rwlock);   /* 获取写锁 */
pthread_rwlock_trywrlock(&rwlock);
/* ... 写操作 ... */
pthread_rwlock_unlock(&rwlock);
```

### 读写锁的公平性问题

默认的读写锁实现中，如果读者一直在读，写者可能**永远拿不到写锁**（写者饥饿）。Linux 的 glibc 读写锁默认是**写者优先**（有写者等待时不再允许新读者进入），可以通过 `pthread_rwlockattr_setkind_np` 调整。

适用场景：读多写少（读:写 > 5:1 时才有明显收益），如配置缓存、路由表、DNS 缓存。

### 带 rwlock 的配置缓存示例

```c
typedef struct { char key[64]; char val[256]; } Config;
static Config g_cfg;
static pthread_rwlock_t g_rwlock = PTHREAD_RWLOCK_INITIALIZER;

/* 任意线程读 */
void get_config(char *key, char *out)
{
    pthread_rwlock_rdlock(&g_rwlock);
    if (strcmp(g_cfg.key, key) == 0) strcpy(out, g_cfg.val);
    pthread_rwlock_unlock(&g_rwlock);
}

/* 只有管理线程写 */
void set_config(const char *key, const char *val)
{
    pthread_rwlock_wrlock(&g_rwlock);
    strncpy(g_cfg.key, key, sizeof(g_cfg.key));
    strncpy(g_cfg.val, val, sizeof(g_cfg.val));
    pthread_rwlock_unlock(&g_rwlock);
}
```

## 7.5 条件变量（pthread_cond）

互斥锁只解决"互斥进入"，解决不了"等待某个条件成立"。条件变量让线程在条件不满足时**高效等待**（不轮询），条件满足时被唤醒。

**底层原理**：`pthread_cond_wait` 原子地做两件事：释放互斥锁 + 睡眠。被唤醒后重新获取互斥锁，再次检查条件。

```
pthread_cond_wait(cond, mutex) 内部：
  1. 把线程加入 cond 的等待队列
  2. 原子地释放 mutex（让其他线程能修改条件）
  3. 睡眠
  4. 被 signal/broadcast 唤醒 → 重新获取 mutex → 返回
```

```c
pthread_cond_t  cond  = PTHREAD_COND_INITIALIZER;
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

/* 等待方（消费者） */
pthread_mutex_lock(&mutex);
while (!condition_met()) {          /* 必须用 while，防止虚假唤醒 */
    pthread_cond_wait(&cond, &mutex);
}
do_work();                          /* 条件成立，处理 */
pthread_mutex_unlock(&mutex);

/* 通知方（生产者） */
pthread_mutex_lock(&mutex);
update_shared_state();
pthread_cond_signal(&cond);         /* 唤醒一个等待者 */
/* pthread_cond_broadcast(&cond);   唤醒所有等待者 */
pthread_mutex_unlock(&mutex);
```

### 为什么 while 循环而不是 if

`pthread_cond_wait` 可能发生**虚假唤醒**（spurious wakeup）：线程被唤醒，但条件仍不满足。这是 POSIX 允许的行为（因为底层基于 futex，某些内核事件可能唤醒等待者）。用 `while` 循环确保唤醒后重新检查条件。

### 带超时的等待

```c
struct timespec deadline;
clock_gettime(CLOCK_REALTIME, &deadline);
deadline.tv_sec += 5;   /* 5 秒超时 */

int rc = pthread_cond_timedwait(&cond, &mutex, &deadline);
if (rc == ETIMEDOUT) {
    printf("等待超时\n");
}
```

### 生产者-消费者完整示例

```c
#define _GNU_SOURCE
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

#define QUEUE_SIZE 8

typedef struct {
    int      buf[QUEUE_SIZE];
    int      head, tail, count;
    pthread_mutex_t mu;
    pthread_cond_t  not_empty;  /* buf 非空时通知消费者 */
    pthread_cond_t  not_full;   /* buf 未满时通知生产者 */
} BoundedQueue;

static BoundedQueue q = {
    .mu        = PTHREAD_MUTEX_INITIALIZER,
    .not_empty = PTHREAD_COND_INITIALIZER,
    .not_full  = PTHREAD_COND_INITIALIZER,
};

void enqueue(int val)
{
    pthread_mutex_lock(&q.mu);
    while (q.count == QUEUE_SIZE)
        pthread_cond_wait(&q.not_full, &q.mu);
    q.buf[q.tail] = val;
    q.tail = (q.tail + 1) % QUEUE_SIZE;
    q.count++;
    pthread_cond_signal(&q.not_empty);
    pthread_mutex_unlock(&q.mu);
}

int dequeue(void)
{
    pthread_mutex_lock(&q.mu);
    while (q.count == 0)
        pthread_cond_wait(&q.not_empty, &q.mu);
    int val = q.buf[q.head];
    q.head = (q.head + 1) % QUEUE_SIZE;
    q.count--;
    pthread_cond_signal(&q.not_full);
    pthread_mutex_unlock(&q.mu);
    return val;
}

static void *producer(void *arg)
{
    for (int i = 0; i < 20; i++) {
        enqueue(i);
        printf("生产: %d\n", i);
    }
    return NULL;
}

static void *consumer(void *arg)
{
    for (int i = 0; i < 20; i++) {
        int val = dequeue();
        printf("消费: %d\n", val);
        usleep(50000);
    }
    return NULL;
}

int main(void)
{
    pthread_t p, c;
    pthread_create(&p, NULL, producer, NULL);
    pthread_create(&c, NULL, consumer, NULL);
    pthread_join(p, NULL);
    pthread_join(c, NULL);
    return 0;
}
```

## 7.6 自旋锁（pthread_spin）

自旋锁与互斥锁的区别：获取不到锁时**忙等**（不放弃 CPU，持续 CAS 循环），而不是睡眠。

```c
pthread_spinlock_t spin;
pthread_spin_init(&spin, PTHREAD_PROCESS_PRIVATE);
pthread_spin_lock(&spin);
/* ... 极短的临界区 ... */
pthread_spin_unlock(&spin);
pthread_spin_destroy(&spin);
```

**什么时候用自旋锁**：

- 临界区极短（< 几十条指令），持锁时间远小于线程上下文切换开销
- 确定不会在持锁时睡眠（持锁睡眠 → 其他线程白白空转，浪费 CPU）
- 多核环境（单核自旋完全是浪费，等待者永远拿不到 CPU）

**不要用自旋锁的情况**：临界区包含 IO、内存分配、条件变量等任何可能阻塞的操作。

## 7.7 POSIX 信号量（线程间）

```c
#include <semaphore.h>
sem_t sem;
sem_init(&sem, 0, 1);   /* pshared=0: 线程间，初始值 1 */
sem_wait(&sem);          /* P: value--, 为 0 则阻塞 */
sem_post(&sem);          /* V: value++, 唤醒一个等待者 */
sem_destroy(&sem);
```

信号量比互斥锁更灵活：初始值可以 > 1（控制并发数），也可以用于"生产者通知消费者"（初始值 0，生产者 post，消费者 wait）。

信号量 vs 条件变量：

| 特性 | 信号量 | 条件变量 |
| --- | --- | --- |
| 有无"历史" | 有（value 计数） | 无（signal 丢失如无等待者） |
| 使用场景 | 限流、资源池 | 等待特定条件 |
| 是否需要配锁 | 否（自带原子性） | 是（必须配 mutex） |

## 7.8 线程局部存储（TLS）

```c
/* 方法 1：__thread（GCC 扩展 / C11 _Thread_local）*/
static __thread int errno_copy;      /* 每个线程有独立副本 */
static __thread char err_buf[256];

/* 方法 2：pthread_key（动态，带析构函数） */
static pthread_key_t err_key;
static pthread_once_t once = PTHREAD_ONCE_INIT;

static void key_init(void) {
    pthread_key_create(&err_key, free);   /* 析构函数：线程退出时 free */
}

char *get_err_buf(void) {
    pthread_once(&once, key_init);
    char *buf = pthread_getspecific(err_key);
    if (!buf) {
        buf = malloc(256);
        pthread_setspecific(err_key, buf);
    }
    return buf;
}
```

**TLS 的典型用途**：

- `errno`（C 库自己就是这么实现的，每个线程有自己的 `errno`）
- 线程专属的错误缓冲、日志上下文、数据库连接
- 避免频繁加锁的线程本地缓存

## 7.9 一次性初始化：pthread_once

```c
static pthread_once_t once_ctrl = PTHREAD_ONCE_INIT;

void init_once(void) {
    /* 只会被执行一次，即使多线程并发调用 pthread_once */
    global_resource = expensive_init();
}

/* 任意线程调用 */
pthread_once(&once_ctrl, init_once);
```

`pthread_once` 常用于**单例模式的懒初始化**：第一个到达的线程执行初始化，后续线程等待完成后直接使用结果。

## 7.10 死锁的产生与避免

死锁的四个必要条件（Coffman 条件）：

- 互斥：资源不能共享
- 持有并等待：已持锁又在等待其他锁
- 不可抢占：已持有的锁不能被强制剥夺
- 循环等待：A 等 B 持有的锁，B 等 A 持有的锁

**避免策略**：

**策略 1：固定加锁顺序**（最简单有效）

```c
/* 所有代码都按 lockA → lockB 的顺序加锁，永远不会 A→B 和 B→A 交叉 */
pthread_mutex_lock(&lockA);
pthread_mutex_lock(&lockB);
/* ... */
pthread_mutex_unlock(&lockB);
pthread_mutex_unlock(&lockA);
```

**策略 2：trylock + 回退**

```c
while (1) {
    pthread_mutex_lock(&lockA);
    if (pthread_mutex_trylock(&lockB) == 0) break;  /* 拿到就退出循环 */
    pthread_mutex_unlock(&lockA);       /* 没拿到 B，释放 A，稍后重试 */
    usleep(rand() % 1000);             /* 随机退避，打破对称 */
}
/* 持有 A 和 B */
pthread_mutex_unlock(&lockB);
pthread_mutex_unlock(&lockA);
```

**策略 3：减少锁的持有时长**

```c
/* 坏 */
pthread_mutex_lock(&mu);
result = compute_heavy_thing();   /* 长时间持锁 */
pthread_mutex_unlock(&mu);

/* 好：计算完再锁 */
result = compute_heavy_thing();
pthread_mutex_lock(&mu);
save_result(result);
pthread_mutex_unlock(&mu);
```

## 7.11 原子操作（C11 / GCC 内建）

对于计数器等简单共享变量，不需要互斥锁，用原子操作更快：

```c
#include <stdatomic.h>

static atomic_long counter = ATOMIC_VAR_INIT(0);

/* 原子递增 */
atomic_fetch_add(&counter, 1);   /* 等同 counter++ 但线程安全 */

/* 原子读 */
long val = atomic_load(&counter);

/* 原子写 */
atomic_store(&counter, 0);

/* CAS（Compare And Swap） */
long expected = 5;
atomic_compare_exchange_strong(&counter, &expected, 10);
/* 若 counter == 5，则设为 10，返回 true；否则把 counter 当前值写回 expected */
```

**内存顺序（memory_order）**：原子操作默认 `memory_order_seq_cst`（顺序一致，最强），有性能开销。常见放宽：

| 内存顺序 | 含义 |
| --- | --- |
| `memory_order_relaxed` | 只保证操作本身原子，不限制其他内存访问顺序 |
| `memory_order_acquire` | 本线程后续操作不会被重排到此操作之前（读屏障） |
| `memory_order_release` | 本线程之前的操作不会被重排到此操作之后（写屏障） |
| `memory_order_acq_rel` | 同时 acquire + release |
| `memory_order_seq_cst` | 全局顺序一致，最强，默认值 |

简单计数器用 `relaxed` 即可；锁的实现（lock → critical section → unlock）需要 `release`/`acquire` 对。

## 7.12 线程属性：栈大小、亲和性

```c
pthread_attr_t attr;
pthread_attr_init(&attr);

/* 设置栈大小（默认 8 MB，嵌入式可能要缩小） */
pthread_attr_setstacksize(&attr, 128 * 1024);   /* 128 KB */

/* 设置 CPU 亲和性 */
cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(0, &cpuset);   /* 绑定到 CPU0 */
pthread_attr_setaffinity_np(&attr, sizeof(cpuset), &cpuset);

/* 设置实时调度策略 */
pthread_attr_setschedpolicy(&attr, SCHED_FIFO);
struct sched_param sp = { .sched_priority = 50 };
pthread_attr_setschedparam(&attr, &sp);
pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED);  /* 不继承，显式设置 */

pthread_create(&tid, &attr, func, arg);
pthread_attr_destroy(&attr);
```

## 7.13 易错点汇总

- **条件变量 wait 用 if 而不是 while**：虚假唤醒导致条件未满足就继续执行
- **在持锁期间调用 pthread_cond_wait 但锁不对**：条件变量的 wait 必须配合 **同一把** mutex
- **unlock 和 signal 的顺序**：先 unlock 再 signal（减少消费者被唤醒后立刻再次阻塞等 mutex）
- **栈上分配大数组**：默认栈 8 MB，线程栈默认也是 8 MB，递归深或局部大数组会栈溢出（SIGSEGV）
- **POSIX sem_init 的 pshared 参数**：线程间填 0，进程间填 1，填错了跨进程用无效
- **pthread_cancel 后没 cleanup_handler**：mutex 被持有时线程被取消，其他线程死锁
- **静态变量在多线程里做初始化**：C++11 之前不是线程安全的，用 `pthread_once` 或 `call_once`
- **atomic_fetch_add 和 ++ 混用**：对 `atomic_long` 直接用 `++` 是未定义行为（UB），必须用 `atomic_fetch_add`
- **读写锁写者饥饿**：读流量很高时写者长时间阻塞，需检查 rwlock 的公平性属性
- **自旋锁持锁时进入条件变量 wait**：持自旋锁睡眠，其他线程自旋白转，性能灾难
