# 第 4 章 进程

## 4.1 进程是什么

进程是**操作系统分配资源的基本单位**。从内核视角看，一个进程就是一个 `task_struct`，包含：

- **地址空间**（`mm_struct`）：代码段、数据段、栈、堆、mmap 区的虚拟地址映射
- **文件描述符表**（`files_struct`）：打开的文件、socket、管道
- **信号处理表**（`sighand_struct`）
- **凭证**（`cred`）：UID/GID/capability
- **调度信息**：优先级、CPU 时间片、亲和性
- **命名空间**（`nsproxy`）：PID/网络/挂载/UTS 等隔离，容器的基础

线程和进程在内核里都是 `task_struct`，区别只在于是否**共享地址空间**：`fork` 创建新地址空间，`clone(CLONE_VM)` 共享地址空间（就是线程）。

## 4.2 进程的内存布局

```
高地址
┌─────────────────────────────┐  0xFFFFFFFF (32位) / 0x7FFFFFFFFFFF (64位)
│  内核空间（用户态不可访问）  │
├─────────────────────────────┤
│  栈（向下增长）              │  局部变量、函数调用帧
│  ↓                          │
│  …… 随机增长 ……              │
│  ↑                          │
│  堆（向上增长）              │  malloc/free
├─────────────────────────────┤
│  mmap 区                    │  动态库、文件映射、匿名大内存
├─────────────────────────────┤
│  .bss（未初始化全局/静态）   │  全零，不占磁盘空间
│  .data（已初始化全局/静态）  │
│  .rodata（只读数据、字符串） │
│  .text（代码段）             │
└─────────────────────────────┘  0x08048000 (32位) / 0x400000 (64位，PIE前)
低地址
```

**栈随机化**（ASLR）：内核启动时随机化栈基址、mmap 基址、堆基址，防止缓冲区溢出利用。可通过 `/proc/sys/kernel/randomize_va_space` 查看（0=关，1=部分，2=全开）。

查看进程内存映射：

```bash
cat /proc/$(pidof bash)/maps
pmap -x $(pidof bash)
```

## 4.3 fork —— 创建子进程

```c
#include <unistd.h>
pid_t fork(void);
/* 父进程返回子进程 PID（> 0），子进程返回 0，失败返回 -1 */
```

### fork 的内核实现原理

`fork` 调用后，内核做的事情（简化）：

```
fork() syscall
  └─ copy_process()
       ├─ dup_task_struct()          // 复制 task_struct 和内核栈
       ├─ copy_mm()                  // 复制内存描述符，但不复制页表项
       │    └─ dup_mmap()
       │         └─ 所有 VMA 标记为 COW（Copy-On-Write）
       │            父子共享同一批物理页，都设为只读
       ├─ copy_files()               // 复制文件描述符表（fd 引用计数 +1）
       ├─ copy_sighand()             // 复制信号处理器
       ├─ copy_creds()               // 复制凭证
       ├─ 分配新的 PID
       └─ 把新 task_struct 加入调度队列
```

**COW（写时拷贝）的效率**：`fork` 之后父子进程共享同一批物理内存页，都标为只读。任何一方尝试写某一页时，触发缺页中断，内核才真正复制那一页，给写方一份私有副本。这使 `fork` + `exec` 的开销极小——大多数情况下子进程立刻 exec，根本不写那些页，物理内存从未被复制。

### fork 后的执行顺序

`fork` 之后父子进程哪个先跑，**由调度器决定**，不确定。历史上父进程先跑更常见，但 Linux 3.4 以后默认子进程先跑（vfork 语义更友好），可通过 `/proc/sys/kernel/sched_child_runs_first` 配置。

**不要在 fork 后同时在父子里写同一个 FILE***：`FILE` 里有用户态缓冲区，fork 会复制这个缓冲区，导致数据被写两份。

### 标准的 fork 模板

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>
#include <errno.h>
#include <string.h>

int main(void)
{
    /* fork 前 fflush，防止缓冲区被复制后重复输出 */
    fflush(stdout);

    pid_t pid = fork();
    if (pid < 0) {
        fprintf(stderr, "fork: %s\n", strerror(errno));
        return EXIT_FAILURE;
    }

    if (pid == 0) {
        /* ---- 子进程 ---- */
        printf("子进程 PID=%d, 父PID=%d\n", getpid(), getppid());
        exit(0);       /* 子进程用 exit 或 _exit，不要 return main */
    }

    /* ---- 父进程 ---- */
    printf("父进程 PID=%d, 子PID=%d\n", getpid(), pid);

    int status;
    pid_t wpid = waitpid(pid, &status, 0);
    if (wpid < 0) { perror("waitpid"); return 1; }

    if (WIFEXITED(status))
        printf("子进程正常退出，退出码 %d\n", WEXITSTATUS(status));
    else if (WIFSIGNALED(status))
        printf("子进程被信号 %d 杀死\n", WTERMSIG(status));

    return EXIT_SUCCESS;
}
```

## 4.4 exec 家族 —— 替换进程映像

`exec` 系列函数用一个新程序**完全替换**当前进程的代码、数据、堆、栈——PID 不变，但进程映像彻底换了。

```c
#include <unistd.h>
int execve(const char *path, char *const argv[], char *const envp[]); /* 基础 syscall */
int execvp(const char *file, char *const argv[]);  /* 在 PATH 里搜索 file */
int execv (const char *path, char *const argv[]);
int execlp(const char *file, const char *arg, ...);  /* 可变参数，NULL 结尾 */
int execl (const char *path, const char *arg, ...);
int execle(const char *path, const char *arg, ..., char *const envp[]);
/* execveat：通过 dirfd 相对路径，支持 AT_EMPTY_PATH 替换 fd 指向的程序 */
int execveat(int dirfd, const char *path, char *const argv[],
             char *const envp[], int flags);
```

### exec 之后什么被保留、什么被清除

| 属性 | exec 后保留 | exec 后清除 |
| --- | --- | --- |
| PID / PPID | 保留 | — |
| 打开的 fd（无 O_CLOEXEC） | 保留 | 有 O_CLOEXEC 的 fd 被关闭 |
| 信号掩码（sigmask） | 保留 | — |
| 信号处理器 | — | 重置为默认 |
| 地址空间（代码/堆/栈） | — | 全部替换 |
| 内存映射 | — | 全部替换 |
| 进程凭证（UID/GID） | 保留（SetUID 文件例外） | — |
| 会话 / 进程组 | 保留 | — |

### 内核调用链（execve）

```
execve(path, argv, envp)
  └─ do_execveat_common()
       ├─ open_exec()               // 打开可执行文件，检查权限
       ├─ bprm_fill_uid()           // 处理 SetUID/SetGID
       ├─ search_binary_handler()   // 找能解析这种格式的 handler
       │    ├─ ELF handler: load_elf_binary()
       │    │    ├─ 读 ELF header，检查 e_type / e_machine
       │    │    ├─ mmap PT_LOAD 段到虚拟地址
       │    │    ├─ 若有动态链接器（PT_INTERP）：mmap ld.so
       │    │    └─ 设置入口点：有 ld.so 则跳 ld.so，否则直接跳 e_entry
       │    └─ 脚本 handler: 解析 #! 行，重新 exec 解释器
       └─ 清除旧地址空间，切换到新映射，跳转到入口点
```

### fork + exec 模板（Shell 执行命令的原理）

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

void run(const char *prog, char *const args[])
{
    pid_t pid = fork();
    if (pid < 0) { perror("fork"); exit(1); }
    if (pid == 0) {
        execvp(prog, args);   /* 失败才会继续往下跑 */
        perror("execvp");
        _exit(127);           /* exec 失败，子进程用 _exit（不冲刷 stdio） */
    }
    int st;
    waitpid(pid, &st, 0);
    printf("%s 退出码: %d\n", prog, WEXITSTATUS(st));
}

int main(void)
{
    char *args[] = { "ls", "-lh", "/tmp", NULL };
    run("ls", args);
    return 0;
}
```

## 4.5 wait / waitpid / waitid —— 回收子进程

子进程退出后，内核保留其 `task_struct`（变成"僵尸进程"）直到父进程 `wait`。如果父进程从不 wait，僵尸进程会一直占用 PID 表项（但不占内存），直到父进程也退出（此时 init/systemd 接管并 wait）。

```c
#include <sys/wait.h>
pid_t wait  (int *status);                      /* 等任意子进程 */
pid_t waitpid(pid_t pid, int *status, int opts);
/* pid > 0 等指定子，pid == -1 等任意子，pid == 0 等同进程组，pid < -1 等 |pid| 进程组 */
/* opts: WNOHANG（非阻塞）| WUNTRACED（也报告被 STOP 的）| WCONTINUED */
int waitid(idtype_t idtype, id_t id, siginfo_t *infop, int opts);
```

### status 解码宏

```c
WIFEXITED(status)    /* 是否正常退出（exit/return） */
WEXITSTATUS(status)  /* 正常退出时的退出码（低 8 位） */
WIFSIGNALED(status)  /* 是否被信号杀死 */
WTERMSIG(status)     /* 杀死进程的信号编号 */
WCOREDUMP(status)    /* 是否产生了 core dump */
WIFSTOPPED(status)   /* 是否被停止（SIGSTOP/SIGTSTP，配合 WUNTRACED） */
WSTOPSIG(status)     /* 停止的信号编号 */
```

### 防止僵尸进程的两种策略

**策略 1：父进程主动 wait**（适合少量子进程）

```c
waitpid(-1, NULL, WNOHANG);  /* 非阻塞回收，放在 SIGCHLD 处理器里 */
```

**策略 2：双重 fork（孤儿化）**

```c
pid_t p1 = fork();
if (p1 == 0) {
    /* 第一个子进程再 fork */
    pid_t p2 = fork();
    if (p2 == 0) {
        /* 孙子进程：父（p1）马上退出，孙子成为孤儿，由 init 接管 */
        /* init 会 wait 它，不会变僵尸 */
        do_real_work();
        _exit(0);
    }
    _exit(0);  /* 第一个子进程立即退出 */
}
waitpid(p1, NULL, 0);  /* 父进程只需 wait 第一个子，孙子由 init 负责 */
```

**策略 3：`signal(SIGCHLD, SIG_IGN)`（Linux 特有）**

```c
signal(SIGCHLD, SIG_IGN);   /* 告诉内核：不关心子进程退出，直接回收，不产生僵尸 */
```

## 4.6 进程的退出

```c
#include <stdlib.h>
#include <unistd.h>
void exit  (int status);   /* 冲刷 stdio 缓冲、调用 atexit 注册的函数、关文件、然后 _exit */
void _exit (int status);   /* 直接进内核，不做任何用户态清理 */
void _Exit (int status);   /* C99，等同 _exit */
int  atexit(void (*fn)(void));   /* 注册退出时的清理函数，最多 32 个，LIFO 顺序调用 */
```

**子进程里要用 `_exit` 而不是 `exit`**：`exit` 会冲刷 stdio 缓冲，而 fork 之后父子共享缓冲区内容，会导致数据被输出两次。`_exit` 跳过清理直接进内核。

## 4.7 守护进程（daemon）

守护进程的要求：

- 脱离终端（不受 SIGHUP）
- 脱离进程组（不受前台进程组的信号）
- 改变工作目录到 `/`（防止占用挂载点）
- 关闭所有继承的 fd
- 标准 IO 重定向到 `/dev/null`

### 手动实现

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/stat.h>
#include <fcntl.h>

void daemonize(void)
{
    /* 第一次 fork：脱离终端，父进程退出 */
    pid_t pid = fork();
    if (pid < 0) { perror("fork"); exit(1); }
    if (pid > 0) exit(0);  /* 父进程退出 */

    /* 创建新会话：子进程成为会话首进程，脱离控制终端 */
    if (setsid() < 0) { perror("setsid"); exit(1); }

    /* 第二次 fork：防止重新获得控制终端（非会话首进程不能获取终端） */
    pid = fork();
    if (pid < 0) { perror("fork"); exit(1); }
    if (pid > 0) exit(0);

    /* 改变工作目录 */
    chdir("/");

    /* 设置 umask */
    umask(0);

    /* 关闭所有 fd 并重定向 0/1/2 到 /dev/null */
    int fd = open("/dev/null", O_RDWR);
    dup2(fd, STDIN_FILENO);
    dup2(fd, STDOUT_FILENO);
    dup2(fd, STDERR_FILENO);
    if (fd > STDERR_FILENO) close(fd);

    /* 关闭其他 fd（简单做法：遍历到 sysconf(_SC_OPEN_MAX)） */
    int maxfd = (int)sysconf(_SC_OPEN_MAX);
    for (int i = STDERR_FILENO + 1; i < maxfd; i++) close(i);
}
```

### 使用 daemon(3)（glibc 提供）

```c
#include <unistd.h>
int daemon(int nochdir, int noclose);
/* nochdir=0 则 chdir("/")，noclose=0 则重定向 0/1/2 到 /dev/null */
daemon(0, 0);
```

### systemd 服务（现代推荐做法）

在 systemd 系统上，不需要手动 daemonize。直接写普通程序，由 systemd 负责管理（进程组、重定向、重启策略）：

```ini
[Unit]
Description=My Linux App
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/myapp
Restart=on-failure
User=myuser
WorkingDirectory=/var/lib/myapp

[Install]
WantedBy=multi-user.target
```

## 4.8 进程优先级与调度

Linux 有两个维度控制调度：**调度策略** 和 **优先级**。

### 普通进程：nice 值

```c
#include <unistd.h>
int nice(int inc);              /* 调整 nice 值，范围 -20（最高）到 +19（最低） */
int getpriority(int which, id_t who);
int setpriority(int which, id_t who, int prio);
/* which: PRIO_PROCESS / PRIO_PGRP / PRIO_USER */
```

nice 值影响内核 CFS 调度器分配的 CPU 时间权重：nice -20 比 nice +19 约多 10 倍的 CPU 时间。

```bash
nice -n 10 ./myapp       # 以 nice=10 启动
renice -n 5 -p 1234      # 修改运行中的进程
```

### 实时进程：调度策略

```c
#include <sched.h>
int sched_setscheduler(pid_t pid, int policy, const struct sched_param *param);
```

| policy | 含义 |
| --- | --- |
| `SCHED_OTHER` | 默认 CFS，nice 值有效 |
| `SCHED_FIFO` | 实时，先进先出，不被抢占（除非更高优先级） |
| `SCHED_RR` | 实时，轮转，同优先级间有时间片 |
| `SCHED_DEADLINE` | 基于 deadline 的实时调度（Linux 3.14+） |
| `SCHED_IDLE` | 比 nice +19 还低，背景任务 |
| `SCHED_BATCH` | CFS 变体，适合批处理，更少调度唤醒 |

实时进程的优先级（rt_priority）范围 1~99，数字越大越高。需要 `CAP_SYS_NICE` 权限或 root。

```c
struct sched_param sp = { .sched_priority = 50 };
sched_setscheduler(0, SCHED_FIFO, &sp);
```

> 嵌入式 Linux 常用 `SCHED_FIFO` 做控制循环（如 PID 控制器），保证响应实时性。

## 4.9 进程的资源限制：getrlimit / setrlimit

```c
#include <sys/resource.h>
int getrlimit(int resource, struct rlimit *rlim);
int setrlimit(int resource, const struct rlimit *rlim);
/* rlim_cur = 软限制（可提升到硬限制），rlim_max = 硬限制 */
```

| resource | 含义 |
| --- | --- |
| `RLIMIT_NOFILE` | 最大打开 fd 数（默认 1024） |
| `RLIMIT_NPROC` | 最大子进程数 |
| `RLIMIT_STACK` | 栈大小上限（默认 8 MB） |
| `RLIMIT_AS` | 虚拟地址空间上限 |
| `RLIMIT_DATA` | 数据段上限 |
| `RLIMIT_CORE` | core dump 大小上限（0 = 不生成） |
| `RLIMIT_CPU` | CPU 时间上限（秒），超出发 SIGXCPU |
| `RLIMIT_FSIZE` | 文件大小上限，超出发 SIGXFSZ |
| `RLIMIT_MEMLOCK` | 可 mlock 的内存量（实时进程需要） |

**嵌入式常见操作**：提高 fd 上限（做高并发服务器时）

```c
struct rlimit rl;
getrlimit(RLIMIT_NOFILE, &rl);
rl.rlim_cur = rl.rlim_max;   /* 提升软限制到硬限制 */
setrlimit(RLIMIT_NOFILE, &rl);
```

## 4.10 /proc 文件系统中的进程信息

`/proc/<pid>/` 是内核暴露进程信息的虚拟文件系统，常用文件：

| 文件 | 内容 |
| --- | --- |
| `status` | 可读的进程状态（VmRSS、线程数、UID 等） |
| `maps` | 内存映射（虚拟地址区间、权限、文件名） |
| `smaps` | 比 maps 更详细（每段的 RSS、PSS、Dirty 等） |
| `fd/` | 目录，每个条目是一个 fd 的符号链接 |
| `fdinfo/<fd>` | fd 的 flags、pos 等 |
| `cmdline` | 启动命令行（null 分隔） |
| `environ` | 环境变量（null 分隔） |
| `stat` | 一行数字，包含 PID、状态、CPU 时间等（ps 读这个） |
| `statm` | 内存使用（页为单位） |
| `oom_score` | OOM killer 打分，分越高越先被杀 |
| `oom_adj` / `oom_score_adj` | 调整 OOM 打分（-1000=永不杀，+1000=优先杀） |
| `limits` | 进程的 rlimit 设置 |
| `wchan` | 进程在内核中等待的符号（睡眠原因） |

```bash
# 查看进程内存使用
cat /proc/1234/status | grep Vm
# 查看进程打开的文件
ls -la /proc/1234/fd
# 查看进程启动命令
cat /proc/1234/cmdline | tr '\0' ' '
```

## 4.11 孤儿进程与僵尸进程

| 概念 | 产生原因 | 影响 | 处理方法 |
| --- | --- | --- | --- |
| 孤儿进程 | 父进程先于子进程退出 | 无害，被 init 接管并最终 wait | 无需处理 |
| 僵尸进程 | 子进程退出但父进程未 wait | 占用 PID，大量时影响新进程创建 | 父进程 wait，或 signal(SIGCHLD, SIG_IGN) |

## 4.12 vfork —— 已过时但值得理解

```c
pid_t vfork(void);
```

`vfork` 比 `fork` 更激进：子进程**直接共享父进程的地址空间**（不做 COW，也不复制页表），父进程**阻塞**直到子进程调用 `exec` 或 `_exit`。目的是为了在 COW 不存在的时代节省内存。

现在 COW 已经让 `fork+exec` 足够快，`vfork` 有严格的使用限制（子进程里不能修改任何变量，不能调用 `exit`），基本不推荐使用，了解原理即可。

## 4.13 进程组、会话与控制终端

这三个概念是理解 Shell 作业控制（`fg/bg/Ctrl-C`）的基础：

```
会话（Session）
  └─ 会话首进程（Session Leader）= shell
  └─ 前台进程组（Foreground Process Group）  ← 控制终端信号（Ctrl-C = SIGINT）发给整组
  └─ 后台进程组 1（bg &）
  └─ 后台进程组 2（bg &）
  └─ 控制终端（/dev/pts/0）
```

```c
#include <unistd.h>
pid_t getpgrp(void);              /* 获取当前进程组 ID */
int   setpgid(pid_t pid, pid_t pgid);  /* 设置进程组 */
pid_t getsid(pid_t pid);          /* 获取会话 ID */
pid_t setsid(void);               /* 创建新会话（脱离控制终端） */
int   tcsetpgrp(int fd, pid_t pgid);   /* 把进程组设为前台 */
```

## 4.14 易错点汇总

- **fork 后双方都 fclose 同一个 FILE***：导致底层 fd 被关两次，第二次 close 可能关掉别的 fd
- **子进程用 exit 而不是 _exit**：stdout 缓冲会被冲刷，如果父进程也有同样内容，输出两遍
- **exec 失败没有 _exit**：exec 失败后继续跑，子进程和父进程都在执行相同逻辑
- **wait 的 status 未初始化就判断**：如果 wait 返回 -1，status 是垃圾值
- **多次 fork 形成 fork bomb**：忘记 pid==0 判断，父子都 fork，指数级爆炸
- **守护进程没做第二次 fork**：会话首进程可以重新获取控制终端，导致收到 SIGHUP
- **实时进程跑死循环占满 CPU**：SCHED_FIFO 不被抢占，会 hang 整个系统，必须加 sched_yield 或超时
- **rlimit 的 RLIMIT_NOFILE 没提前调大**：高并发服务 accept 新连接时突然 EMFILE，排查困难
