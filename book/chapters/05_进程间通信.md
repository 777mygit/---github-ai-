# 第 5 章 进程间通信（IPC）

## 5.1 为什么需要 IPC，以及怎么选

进程拥有独立的地址空间，天然无法直接读写对方的变量。Linux 提供了多种 IPC 机制，各有适用场景：

| 机制 | 数据流向 | 有无名字 | 跨主机 | 适合场景 |
| --- | --- | --- | --- | --- |
| 匿名管道 pipe | 单向 | 无（只在亲缘进程间） | 否 | 父子进程流数据 |
| 命名管道 FIFO | 单向 | 有（文件系统路径） | 否 | 无亲缘进程流数据 |
| 信号 signal | 单向（通知） | — | 否 | 异步事件通知 |
| POSIX 消息队列 | 双向、带优先级 | 有（/mq_name） | 否 | 结构化消息传递 |
| System V 消息队列 | 双向、带类型 | 有（key_t） | 否 | 遗留系统兼容 |
| 共享内存 | 共享读写 | 有 | 否 | 最高带宽，需配锁 |
| Unix 域 socket | 双向、流/数据报 | 有（路径）或匿名 | 否 | 通用高性能本机 IPC |
| TCP/UDP socket | 双向 | IP + 端口 | 是 | 跨主机通信 |
| eventfd | 计数通知 | 无 | 否 | 线程/进程间事件 |
| signalfd | 信号转 fd | — | 否 | 用 epoll 统一处理信号 |

**选型口诀**：

- 父子流数据 → **pipe**
- 无亲缘流数据 → **FIFO** 或 **Unix socket**（socket 更灵活）
- 最高带宽 → **共享内存** + 信号量同步
- 需要优先级/消息边界 → **POSIX 消息队列**
- 跨主机 → **socket**

## 5.2 匿名管道（pipe）

### 原理

```c
#include <unistd.h>
int pipe (int pipefd[2]);        /* pipefd[0]=读端, pipefd[1]=写端 */
int pipe2(int pipefd[2], int flags);  /* flags: O_NONBLOCK | O_CLOEXEC */
```

管道是内核维护的一块**循环缓冲区**（默认 64 KB，可通过 `/proc/sys/fs/pipe-max-size` 查看上限）。写端写入数据，读端读取，数据只能单向流动。

```
写端 pipefd[1] ──────────► [内核循环缓冲区 64KB] ──────────► 读端 pipefd[0]
```

**关键行为**：

- 读端没有数据时 `read` 阻塞（写端全部关闭后 `read` 返回 0，即 EOF）
- 写端写满（缓冲区满）时 `write` 阻塞（读端全部关闭后 `write` 收到 SIGPIPE 或返回 EPIPE）
- 写入 ≤ `PIPE_BUF`（4096 字节）的单次 `write` 是**原子**的，多个写端并发不会交错

### 内核调用链

```
pipe2(fds, O_CLOEXEC)
  └─ do_pipe2()
       ├─ create_pipe_files()
       │    ├─ alloc_file_pseudo()   // 在 pipefs 上创建两个 file 对象
       │    └─ 分配 pipe_inode_info（内核循环缓冲区）
       └─ fd_install() × 2          // 把读端/写端 file* 装入 fd 表
```

### 标准用法：父子进程通信

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/wait.h>

int main(void)
{
    int fds[2];
    if (pipe2(fds, O_CLOEXEC) < 0) { perror("pipe2"); exit(1); }

    pid_t pid = fork();
    if (pid < 0) { perror("fork"); exit(1); }

    if (pid == 0) {
        /* 子进程：只写，关闭读端 */
        close(fds[0]);
        const char *msg = "hello from child";
        write(fds[1], msg, strlen(msg));
        close(fds[1]);
        _exit(0);
    }

    /* 父进程：只读，关闭写端 */
    close(fds[1]);          /* 必须关写端！否则永远读不到 EOF */
    char buf[256] = {0};
    ssize_t n = read(fds[0], buf, sizeof(buf) - 1);
    printf("父进程收到: %.*s\n", (int)n, buf);
    close(fds[0]);
    waitpid(pid, NULL, 0);
    return 0;
}
```

> **最常见的 bug**：fork 后父进程没关写端，读端永远阻塞（因为写端还有一个打开的引用）。

### popen / pclose

```c
FILE *popen(const char *command, const char *type);  /* type: "r" 或 "w" */
int   pclose(FILE *stream);
```

`popen` 是 `pipe + fork + exec + fdopen` 的组合，让你像读写文件一样读写子进程的 stdout/stdin。`pclose` 会 `waitpid` 回收子进程。适合简单场景，不适合需要同时读写或精确错误处理的场合。

## 5.3 命名管道（FIFO）

```c
#include <sys/stat.h>
int mkfifo(const char *path, mode_t mode);
int mkfifoat(int dirfd, const char *path, mode_t mode);
```

FIFO 是一个特殊文件（`ls -l` 看到类型 `p`），无亲缘进程通过路径打开后就能像 pipe 一样通信。

**阻塞语义**：
- `open(fifo, O_RDONLY)` 会阻塞，直到另一端 `open(..., O_WRONLY)` 也打开
- `open(fifo, O_RDONLY | O_NONBLOCK)` 不等写端，立刻返回

```bash
mkfifo /tmp/myfifo
# 终端 1：
cat < /tmp/myfifo
# 终端 2：
echo "hello fifo" > /tmp/myfifo
```

程序里的典型用法：

```c
/* 服务端 */
mkfifo("/tmp/req.fifo", 0600);
int rfd = open("/tmp/req.fifo", O_RDONLY);

/* 客户端 */
int wfd = open("/tmp/req.fifo", O_WRONLY);
write(wfd, request, len);
```

## 5.4 System V IPC 三件套

System V IPC（消息队列 / 共享内存 / 信号量）是 Unix 遗留接口，API 繁琐，但在嵌入式 Linux 和遗留系统里仍然常见，要能看懂。

### key_t —— IPC 对象的命名

```c
#include <sys/ipc.h>
key_t ftok(const char *path, int proj_id);
/* 用文件 inode + proj_id 生成 key，不同进程只要用同一 path+proj_id 就能拿到同一 key */
```

也可以直接用 `IPC_PRIVATE`（只用于有亲缘关系的进程）。

### System V 消息队列

```c
#include <sys/msg.h>
int   msgget(key_t key, int msgflg);          /* 创建/获取队列 */
int   msgsnd(int msqid, const void *msg, size_t msgsz, int msgflg);
ssize_t msgrcv(int msqid, void *msg, size_t msgsz, long msgtyp, int msgflg);
int   msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

消息必须以 `long mtype` 开头：

```c
struct mymsg {
    long mtype;   /* 必须 > 0，接收时可按类型过滤 */
    char text[256];
};
```

`msgtyp` 的过滤规则：

| msgtyp 值 | 效果 |
| --- | --- |
| 0 | 读取队列里最早的任意消息 |
| > 0 | 读取 mtype == msgtyp 的最早消息 |
| < 0 | 读取 mtype ≤ |msgtyp| 里最小类型的消息 |

### System V 共享内存

```c
#include <sys/shm.h>
int  shmget(key_t key, size_t size, int shmflg);    /* 创建/获取 */
void *shmat(int shmid, const void *shmaddr, int shmflg);  /* 映射到当前进程 */
int  shmdt(const void *shmaddr);                    /* 取消映射 */
int  shmctl(int shmid, int cmd, struct shmid_ds *buf);    /* IPC_RMID 删除 */
```

**System V 共享内存 vs mmap(MAP_SHARED)**：

| 特性 | System V shm | mmap(MAP_SHARED) |
| --- | --- | --- |
| 命名 | key_t（ftok） | 文件路径 或 shm_open |
| 生命周期 | 进程退出后仍然存在，须显式 IPC_RMID | 文件存在就存在，匿名映射随进程消亡 |
| 可见性 | ipcs -m 可查 | 普通文件 |
| 推荐度 | 遗留 | 现代推荐 |

### System V 信号量

```c
#include <sys/sem.h>
int semget(key_t key, int nsems, int semflg);
int semop (int semid, struct sembuf *sops, size_t nsops);
int semctl(int semid, int semnum, int cmd, ...);

struct sembuf {
    unsigned short sem_num;  /* 信号量索引 */
    short          sem_op;   /* >0 释放，<0 申请，0 等待归零 */
    short          sem_flg;  /* IPC_NOWAIT | SEM_UNDO */
};
```

`SEM_UNDO` 让进程退出时自动撤销对信号量的操作——强烈推荐加上，防止进程崩溃导致信号量永久被锁。

## 5.5 POSIX IPC（推荐使用）

POSIX IPC 是比 System V 更现代、更简洁的接口，以文件路径（`/name` 形式）命名，在 `/dev/shm` 或 `/run/shm` 下有对应文件，可用 `ls` 查看。

### POSIX 共享内存

```c
#include <sys/mman.h>
#include <fcntl.h>
int shm_open  (const char *name, int oflag, mode_t mode);
int shm_unlink(const char *name);
/* 链接时加 -lrt */
```

```c
/* 创建方 */
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0600);
ftruncate(fd, 4096);
void *p = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
close(fd);  /* mmap 后 fd 可关 */
memcpy(p, "shared data", 11);

/* 使用方（另一进程） */
int fd2 = shm_open("/myshm", O_RDWR, 0);
void *q  = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd2, 0);
printf("%s\n", (char *)q);  /* 读到 "shared data" */

/* 清理 */
munmap(p, 4096);
shm_unlink("/myshm");   /* 删除命名对象 */
```

### POSIX 消息队列

```c
#include <mqueue.h>
mqd_t mq_open  (const char *name, int oflag, mode_t mode, struct mq_attr *attr);
int   mq_close (mqd_t mqdes);
int   mq_unlink(const char *name);
int   mq_send  (mqd_t mqdes, const char *msg, size_t len, unsigned prio);
ssize_t mq_receive(mqd_t mqdes, char *msg, size_t len, unsigned *prio);
int   mq_notify(mqd_t mqdes, const struct sigevent *sev);
/* 链接时加 -lrt */
```

POSIX 消息队列的特点：

- 按**优先级**出队（高优先级先收）
- 可通过 `mq_notify` 注册信号或线程通知，不需要轮询
- `/proc/sys/fs/mqueue/` 可调整队列属性限制

```c
struct mq_attr attr = { .mq_maxmsg = 10, .mq_msgsize = 256 };
mqd_t mq = mq_open("/testq", O_CREAT | O_RDWR, 0600, &attr);

/* 发 */
mq_send(mq, "hello", 5, 1);   /* prio=1 */

/* 收 */
char buf[256];
unsigned prio;
ssize_t n = mq_receive(mq, buf, sizeof(buf), &prio);
printf("收到: %.*s (优先级 %u)\n", (int)n, buf, prio);

mq_close(mq);
mq_unlink("/testq");
```

### POSIX 信号量

```c
#include <semaphore.h>
/* 命名信号量 */
sem_t *sem_open  (const char *name, int oflag, mode_t mode, unsigned value);
int    sem_close (sem_t *sem);
int    sem_unlink(const char *name);
/* 匿名信号量（用于共享内存中的进程间，或同进程线程间） */
int    sem_init  (sem_t *sem, int pshared, unsigned value);
int    sem_destroy(sem_t *sem);
/* 操作 */
int    sem_wait    (sem_t *sem);   /* P 操作，value-- ，为 0 则阻塞 */
int    sem_trywait (sem_t *sem);   /* 非阻塞，失败返回 EAGAIN */
int    sem_timedwait(sem_t *sem, const struct timespec *abs_timeout);
int    sem_post    (sem_t *sem);   /* V 操作，value++ */
int    sem_getvalue(sem_t *sem, int *sval);
```

POSIX 信号量比 System V 信号量更轻量，API 更直观。进程间用命名信号量，线程间用 `sem_init(sem, 0, val)`。

## 5.6 Unix 域 Socket

Unix 域 socket（`AF_UNIX` / `AF_LOCAL`）是**本机进程间最通用的 IPC 机制**：既有消息边界（SOCK_DGRAM）也有流（SOCK_STREAM），还支持传递**文件描述符**（`SCM_RIGHTS`），这是其他 IPC 机制做不到的。

### 流式 Unix socket 示例

```c
/* server.c */
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <sys/socket.h>
#include <sys/un.h>

#define SOCK_PATH "/tmp/echo.sock"

int main(void)
{
    unlink(SOCK_PATH);

    int sfd = socket(AF_UNIX, SOCK_STREAM | SOCK_CLOEXEC, 0);
    struct sockaddr_un addr = { .sun_family = AF_UNIX };
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path) - 1);

    bind(sfd, (struct sockaddr *)&addr, sizeof(addr));
    listen(sfd, 5);
    printf("监听 %s\n", SOCK_PATH);

    int cfd = accept(sfd, NULL, NULL);
    char buf[256];
    ssize_t n = read(cfd, buf, sizeof(buf));
    printf("收到: %.*s\n", (int)n, buf);
    write(cfd, buf, (size_t)n);   /* echo 回去 */
    close(cfd); close(sfd);
    unlink(SOCK_PATH);
    return 0;
}
```

```c
/* client.c */
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <sys/socket.h>
#include <sys/un.h>

#define SOCK_PATH "/tmp/echo.sock"

int main(void)
{
    int fd = socket(AF_UNIX, SOCK_STREAM | SOCK_CLOEXEC, 0);
    struct sockaddr_un addr = { .sun_family = AF_UNIX };
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path) - 1);
    connect(fd, (struct sockaddr *)&addr, sizeof(addr));

    write(fd, "hello unix socket", 17);
    char buf[256];
    ssize_t n = read(fd, buf, sizeof(buf));
    printf("echo: %.*s\n", (int)n, buf);
    close(fd);
    return 0;
}
```

### 通过 Unix socket 传递 fd（SCM_RIGHTS）

这是 Unix domain socket 最强大的特性：进程 A 把一个 fd 发给进程 B，B 拿到的是**独立的 fd**，引用同一个内核 `file` 对象。常见用途：权限分离（低权限进程从高权限进程获取已打开的文件）、容器运行时传递网络命名空间 fd。

```c
/* 发送 fd */
void send_fd(int sock, int fd)
{
    char cmsgbuf[CMSG_SPACE(sizeof(int))];
    struct iovec iov = { .iov_base = "x", .iov_len = 1 };  /* 必须有至少 1 字节数据 */
    struct msghdr msg = {
        .msg_iov = &iov, .msg_iovlen = 1,
        .msg_control = cmsgbuf, .msg_controllen = sizeof(cmsgbuf),
    };
    struct cmsghdr *cmsg = CMSG_FIRSTHDR(&msg);
    cmsg->cmsg_level = SOL_SOCKET;
    cmsg->cmsg_type  = SCM_RIGHTS;
    cmsg->cmsg_len   = CMSG_LEN(sizeof(int));
    memcpy(CMSG_DATA(cmsg), &fd, sizeof(int));
    sendmsg(sock, &msg, 0);
}

/* 接收 fd */
int recv_fd(int sock)
{
    char cmsgbuf[CMSG_SPACE(sizeof(int))];
    char dummy;
    struct iovec iov = { .iov_base = &dummy, .iov_len = 1 };
    struct msghdr msg = {
        .msg_iov = &iov, .msg_iovlen = 1,
        .msg_control = cmsgbuf, .msg_controllen = sizeof(cmsgbuf),
    };
    recvmsg(sock, &msg, 0);
    struct cmsghdr *cmsg = CMSG_FIRSTHDR(&msg);
    int fd;
    memcpy(&fd, CMSG_DATA(cmsg), sizeof(int));
    return fd;
}
```

## 5.7 eventfd —— 轻量级进程/线程间事件通知

```c
#include <sys/eventfd.h>
int eventfd(unsigned int initval, int flags);
/* flags: EFD_NONBLOCK | EFD_CLOEXEC | EFD_SEMAPHORE */
```

`eventfd` 返回一个 fd，可以被 `epoll` 监听：

- `write(fd, &val, 8)`：计数器 += val（`uint64_t`）
- `read(fd, &val, 8)`：读出计数器当前值并清零（`EFD_SEMAPHORE` 模式每次只减 1）

非常适合线程池"任务到达通知"、进程间"有新数据"通知，且可以和 epoll 无缝集成。

```c
int efd = eventfd(0, EFD_CLOEXEC | EFD_NONBLOCK);

/* 生产者（另一个线程/进程）*/
uint64_t one = 1;
write(efd, &one, sizeof(one));   /* 通知有 1 个事件 */

/* 消费者（epoll 告知 efd 可读后）*/
uint64_t cnt;
read(efd, &cnt, sizeof(cnt));    /* cnt = 有多少次通知 */
```

## 5.8 共享内存 + 同步的完整示例

共享内存带宽最高，但必须配合同步机制，否则会产生竞态。下面用**匿名信号量放在共享内存里**实现进程间生产者-消费者：

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <semaphore.h>
#include <sys/mman.h>
#include <sys/wait.h>

#define BUF_SIZE 64
#define N_MSG    5

typedef struct {
    sem_t empty;    /* 空槽信号量 */
    sem_t full;     /* 满槽信号量 */
    sem_t mutex;    /* 互斥 */
    int   head, tail, count;
    char  buf[N_MSG][BUF_SIZE];
} SharedQ;

int main(void)
{
    SharedQ *q = mmap(NULL, sizeof(*q),
                      PROT_READ | PROT_WRITE,
                      MAP_SHARED | MAP_ANONYMOUS, -1, 0);

    sem_init(&q->empty, 1, N_MSG);  /* pshared=1: 进程间可用 */
    sem_init(&q->full,  1, 0);
    sem_init(&q->mutex, 1, 1);

    pid_t pid = fork();
    if (pid == 0) {
        /* 消费者 */
        for (int i = 0; i < N_MSG; i++) {
            sem_wait(&q->full);
            sem_wait(&q->mutex);
            printf("消费: %s\n", q->buf[q->head]);
            q->head = (q->head + 1) % N_MSG;
            q->count--;
            sem_post(&q->mutex);
            sem_post(&q->empty);
        }
        _exit(0);
    }

    /* 生产者 */
    for (int i = 0; i < N_MSG; i++) {
        sem_wait(&q->empty);
        sem_wait(&q->mutex);
        snprintf(q->buf[q->tail], BUF_SIZE, "message-%d", i);
        q->tail = (q->tail + 1) % N_MSG;
        q->count++;
        sem_post(&q->mutex);
        sem_post(&q->full);
        usleep(100000);
    }

    waitpid(pid, NULL, 0);
    sem_destroy(&q->empty);
    sem_destroy(&q->full);
    sem_destroy(&q->mutex);
    munmap(q, sizeof(*q));
    return 0;
}
```

## 5.9 IPC 资源泄漏与清理

System V IPC 对象在进程退出后**默认不删除**，是嵌入式 Linux 中常见的资源泄漏源：

```bash
ipcs -a          # 查看所有 System V IPC 对象（消息队列/共享内存/信号量）
ipcrm -m <shmid> # 删除共享内存
ipcrm -q <msqid> # 删除消息队列
ipcrm -s <semid> # 删除信号量
```

POSIX IPC 对象（`shm_open`/`mq_open`/`sem_open` 创建的）在 `/dev/shm/` 或 `/dev/mqueue/` 下有对应文件：

```bash
ls /dev/shm/     # POSIX 共享内存
ls /dev/mqueue/  # POSIX 消息队列
```

**最佳实践**：在程序启动时用 `atexit` 注册清理函数，或者在 `SIGTERM/SIGINT` 处理器里清理。

## 5.10 易错点汇总

- **pipe 读端没关写端**：父进程既是写者又没关写端，read 永远阻塞
- **pipe 缓冲区写满死锁**：父写子读，如果父在一次 write 里写超 64 KB 而子没在读，双方都阻塞
- **System V IPC key 冲突**：两个不相关的程序碰巧用了相同的 key_t，互相干扰
- **共享内存忘记同步**：直接在共享内存里写结构体而不加锁，多核 CPU 上必然出现数据撕裂
- **POSIX 消息队列消息大于 mq_msgsize**：mq_send 返回 EMSGSIZE，消息截断
- **sem_init 的 pshared 填 0**：想跨进程用，但 pshared=0 是线程间，放在共享内存里无效
- **shm_open 后没 ftruncate**：mmap 后访问超过初始大小的区域触发 SIGBUS
- **Unix socket 文件没 unlink**：重启程序 bind 报 EADDRINUSE，需要在 bind 前 unlink
