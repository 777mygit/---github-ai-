# 第 1 章 开发环境与工具链

## 1.1 开发环境组合

嵌入式 Linux 应用开发通常是**主机**（x86_64 开发机）+ **目标机**（ARM 开发板 / 虚拟机）的双机模式：

| 角色 | 常见平台 | 作用 |
| --- | --- | --- |
| 主机 (host) | Ubuntu 22.04 / WSL2 / macOS | 编辑、编译、交叉编译、版本管理 |
| 目标机 (target) | ARM 开发板、QEMU、x86 Linux VM | 运行可执行文件、抓日志、联调 |
| 连接方式 | SSH / 串口 / NFS / TFTP / ADB | 传文件、执行命令 |

建议在主机上装这些包（Ubuntu）：

```bash
sudo apt install -y build-essential gdb gdb-multiarch \
    gcc-arm-linux-gnueabihf g++-arm-linux-gnueabihf \
    make cmake ninja-build pkg-config \
    git vim tree tmux \
    strace ltrace valgrind
```

## 1.2 GCC 编译流程

从 `.c` 到可执行文件要经过 4 个阶段，GCC 把它们打包成了一条命令，但排错时你必须能手动拆开：

```
源文件 hello.c
   │  预处理 (cpp / gcc -E)        展开 #include / #define / 条件编译
   ▼
hello.i  预处理后 C
   │  编译 (cc1 / gcc -S)          生成汇编
   ▼
hello.s  汇编代码
   │  汇编 (as / gcc -c)           生成目标文件
   ▼
hello.o  目标文件 (ELF relocatable)
   │  链接 (ld / gcc)              合并库、解析符号、生成 ELF
   ▼
hello    可执行文件
```

对应命令：

```bash
gcc -E hello.c -o hello.i     # 预处理
gcc -S hello.i -o hello.s     # 编译
gcc -c hello.s -o hello.o     # 汇编
gcc    hello.o -o hello       # 链接
```

一步到位：

```bash
gcc hello.c -o hello
```

## 1.3 GCC 常用选项速查

| 选项 | 含义 |
| --- | --- |
| `-o <file>` | 指定输出文件名 |
| `-c` | 只编译+汇编，不链接，产出 `.o` |
| `-E` / `-S` | 只跑到预处理 / 汇编阶段 |
| `-g` | 加调试信息，给 GDB 用 |
| `-O0 / -O1 / -O2 / -O3 / -Os` | 优化级别，从关闭到最高，`-Os` 为体积优化 |
| `-Wall -Wextra -Werror` | 开全部警告并将警告升级为错误 |
| `-std=c99 / c11 / gnu11` | 指定 C 标准 |
| `-I<dir>` | 额外头文件搜索路径 |
| `-L<dir>` | 额外库搜索路径 |
| `-l<name>` | 链接 `lib<name>.so` 或 `.a`，如 `-lpthread` |
| `-static` | 静态链接 |
| `-fPIC` | 生成位置无关代码，做动态库必加 |
| `-shared` | 生成动态库（配合 `-fPIC`） |
| `-D<MACRO>[=val]` | 等价于源码 `#define` |
| `-pthread` | 启用 POSIX 线程（比 `-lpthread` 更全，还会开 `_REENTRANT`） |
| `-save-temps` | 保留中间文件 `.i` `.s` `.o` |
| `-MMD -MP` | 生成依赖文件 `.d`，Makefile 里用 |

## 1.4 静态库与动态库

**静态库 `.a`**：编译期把 `.o` 打包进可执行文件，独立运行但体积大。

```bash
gcc -c add.c sub.c
ar rcs libmath.a add.o sub.o
gcc main.c -L. -lmath -o app        # -lmath 找的是 libmath.a 或 libmath.so
```

**动态库 `.so`**：运行期由 `ld.so` 加载，体积小、可升级、多个进程共享。

```bash
gcc -fPIC -c add.c sub.c
gcc -shared add.o sub.o -o libmath.so
gcc main.c -L. -lmath -o app
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
./app
```

查动态库依赖：

```bash
ldd ./app                # 看 app 依赖哪些 .so
nm -D libmath.so         # 看动态符号表
objdump -p libmath.so | grep NEEDED
```

**优先级与查找路径**（从高到低）：

1. 编译时用 `-Wl,-rpath=<path>` 写死的路径
2. 环境变量 `LD_LIBRARY_PATH`
3. `/etc/ld.so.cache`（由 `ldconfig` 从 `/etc/ld.so.conf*` 生成）
4. 默认 `/lib` `/usr/lib` `/lib64` `/usr/lib64`

## 1.5 GDB 调试速查

**编译时必须加 `-g`**，否则源码级调试不可用：

```bash
gcc -g -O0 demo.c -o demo
gdb ./demo
```

常用命令：

| 命令 | 简写 | 作用 |
| --- | --- | --- |
| `run [args]` | `r` | 运行程序 |
| `break <loc>` | `b` | 打断点，`b main`、`b demo.c:42`、`b func` |
| `tbreak` | `tb` | 一次性断点 |
| `watch <expr>` | — | 监视变量，变化时断下 |
| `continue` | `c` | 继续运行 |
| `next` | `n` | 单步，跨过函数 |
| `step` | `s` | 单步，进入函数 |
| `finish` | `fin` | 运行到当前函数返回 |
| `print <expr>` | `p` | 打印表达式，`p/x` 十六进制，`p/t` 二进制 |
| `info locals` | `i lo` | 打印局部变量 |
| `info registers` | `i r` | 打印寄存器 |
| `backtrace` | `bt` | 打印调用栈，`bt full` 带变量 |
| `frame N` | `f N` | 切换到第 N 层栈帧 |
| `list` | `l` | 显示源码 |
| `x/<n><f><u> <addr>` | — | 查内存，如 `x/16xb buf` 看 16 字节 |
| `set var x=10` | — | 运行时改变量 |
| `attach <pid>` | — | 附加到运行中的进程 |
| `core-file core` | — | 分析 core dump |
| `quit` | `q` | 退出 |

**调试多进程/多线程**：

```
(gdb) set follow-fork-mode child    # fork 后跟子进程
(gdb) set detach-on-fork off        # 同时调试父子
(gdb) info threads                  # 列线程
(gdb) thread 3                      # 切到 3 号线程
(gdb) thread apply all bt           # 所有线程打栈
```

**core dump**：

```bash
ulimit -c unlimited                 # 当前 shell 开启 core
echo 'core.%e.%p' | sudo tee /proc/sys/kernel/core_pattern
./demo                              # 崩溃后生成 core.demo.1234
gdb ./demo core.demo.1234
(gdb) bt
```

## 1.6 Makefile 基础

一份最小可用的 Makefile：

```makefile
CC      := gcc
CFLAGS  := -Wall -Wextra -O2 -g
LDFLAGS := -lpthread

SRCS    := $(wildcard *.c)
OBJS    := $(SRCS:.c=.o)
TARGET  := app

.PHONY: all clean

all: $(TARGET)

$(TARGET): $(OBJS)
	$(CC) $(OBJS) -o $@ $(LDFLAGS)

%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

clean:
	rm -f $(OBJS) $(TARGET)
```

要点：

- **规则格式**：`目标: 依赖` 换行后用 **Tab** 开头写命令，不能用空格
- `$@` 目标名，`$<` 第一个依赖，`$^` 全部依赖
- `:=` 立即展开，`=` 延迟展开，`?=` 未定义才赋值，`+=` 追加
- `.PHONY` 声明伪目标，避免同名文件存在时失效

**自动依赖**（头文件改动后正确重编）：

```makefile
CFLAGS += -MMD -MP
DEPS    := $(OBJS:.o=.d)
-include $(DEPS)
```

## 1.7 CMake 基础

CMake 更适合跨平台、跨编译器，CLion / VSCode CMake Tools 也原生支持。

`CMakeLists.txt` 最小模板：

```cmake
cmake_minimum_required(VERSION 3.16)
project(demo LANGUAGES C)

set(CMAKE_C_STANDARD 11)
set(CMAKE_C_STANDARD_REQUIRED ON)
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)   # 给 clangd 用

add_compile_options(-Wall -Wextra -O2 -g)

add_executable(demo main.c utils.c)
target_link_libraries(demo PRIVATE pthread)
```

构建：

```bash
cmake -S . -B build -G Ninja
cmake --build build -j
./build/demo
```

**交叉编译**用 toolchain 文件：

```cmake
# toolchain-arm.cmake
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR arm)
set(CMAKE_C_COMPILER   arm-linux-gnueabihf-gcc)
set(CMAKE_CXX_COMPILER arm-linux-gnueabihf-g++)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
```

```bash
cmake -S . -B build-arm -DCMAKE_TOOLCHAIN_FILE=toolchain-arm.cmake
cmake --build build-arm -j
```

## 1.8 交叉编译

核心概念：

- **build**：跑编译器的机器
- **host**：编译出的程序要跑的机器（常说的"目标机"）
- **target**：编译出的**编译器**将来要编译给谁（只有做"编译编译器"时才有意义）

命名规律：`<arch>-<vendor>-<os>-<abi>`，例如：

- `arm-linux-gnueabihf`：ARM + Linux + glibc + 硬浮点
- `aarch64-linux-gnu`：ARM64 + Linux + glibc
- `mipsel-linux-musl`：MIPS 小端 + Linux + musl libc

常用验证：

```bash
arm-linux-gnueabihf-gcc -print-sysroot
arm-linux-gnueabihf-gcc hello.c -o hello
file hello                          # 确认是 ARM ELF
arm-linux-gnueabihf-readelf -d hello | grep NEEDED
```

## 1.9 常用辅助工具

| 工具 | 用途 | 常用命令 |
| --- | --- | --- |
| `strace` | 跟踪系统调用 | `strace -f -e trace=openat,read ./app` |
| `ltrace` | 跟踪库函数调用 | `ltrace -e 'malloc+free' ./app` |
| `valgrind` | 内存错误与泄漏 | `valgrind --leak-check=full ./app` |
| `perf` | 性能剖析 | `perf record ./app && perf report` |
| `gprof` | 函数级耗时 | 编译加 `-pg`，跑完 `gprof ./app gmon.out` |
| `nm` | 符号表 | `nm -C app` 带 C++ demangle |
| `objdump` | 反汇编 / ELF 内容 | `objdump -d app`、`objdump -h app` |
| `readelf` | ELF 元信息 | `readelf -a app` |
| `size` | 段大小 | `size app` |
| `strip` | 去符号 | `strip app` |
| `file` | 识别文件类型 | `file app` |
| `hexdump` / `xxd` | 看二进制 | `xxd -g 1 file.bin` |

## 1.10 易错点

1. **头文件找不到**：先确认 `-I` 加对了，或用 `gcc -H` 打印头文件搜索过程
2. **链接找不到符号**：C++ 名称粉碎，跟 C 互调要用 `extern "C"`；`-l<name>` 要放在**使用它的 `.o` 之后**
3. **`.so` 运行时找不到**：`ldd` 看缺哪个，再配 `LD_LIBRARY_PATH` 或 `rpath`
4. **Makefile 命令不是 Tab 开头**：`make: *** missing separator`，全是因为空格
5. **调试时变量被优化掉**：编译加 `-O0`，再加 `-g3`
6. **交叉编译的二进制跑不起来**：目标机 libc 版本不匹配，用 `-static` 或同步 sysroot
